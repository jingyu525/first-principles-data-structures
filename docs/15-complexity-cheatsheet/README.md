## 一、时间复杂度（Time Complexity）

按照增长速度排序。

| 复杂度        | 名称   | 100万数据量感觉   |
| ---------- | ---- | ----------- |
| O(1)       | 常数   | 几乎瞬间        |
| O(log n)   | 对数   | 十几二十次操作     |
| O(√n)      | 平方根  | 约1000次      |
| O(n)       | 线性   | 100万次       |
| O(n log n) | 线性对数 | 约2000万次     |
| O(n²)      | 平方   | 10¹² 次，不可接受 |
| O(n³)      | 立方   | 基本无法运行      |
| O(2ⁿ)      | 指数   | 非常恐怖        |
| O(n!)      | 阶乘   | 几乎不能计算      |

## 一页记忆地图（第一性原理）

可以把复杂度归纳为四类思维：

| 操作本质           |                典型复杂度 | 代表算法/数据结构                  |
| -------------- | -------------------: | -------------------------- |
| **直接定位**（无需遍历） |             **O(1)** | 数组下标访问、哈希表查找（平均）、栈/队列顶端操作  |
| **每次排除一半**     |         **O(log n)** | 二分查找、平衡树、堆的插入/删除           |
| **逐个扫描**       |             **O(n)** | 遍历数组、链表查找、BFS/DFS（按节点和边总量） |
| **两层或多层组合**    | **O(n²)、O(n log n)** | 冒泡排序、选择排序、归并排序、快速排序        |

**记忆原则：**

* 能直接找到，就追求 **O(1)**。
* 能缩小搜索空间，就追求 **O(log n)**。
* 至少要看一遍所有数据，就是 **O(n)**。
* 多次遍历或嵌套遍历，通常会上升到 **O(n²)** 或更高。

对于后端开发（尤其是 Go），真正需要熟练掌握的复杂度主要集中在 **O(1)、O(log n)、O(n)、O(n log n)** 四档。绝大多数高性能系统设计，都是通过选择合适的数据结构，把算法从 **O(n²)** 优化到这四种复杂度之一。
