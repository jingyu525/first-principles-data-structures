# 10 B+Tree

## 第一性原理：为什么需要 B+Tree？

### 问题

```
BST / 红黑树 / AVL → 内存中很好

但在磁盘上呢？

磁盘特性：
- 顺序读写快（~100MB/s）
- 随机读写慢（~100 IOPS）
- 每次 IO 至少读一个「页」（通常 16KB）

BST 在磁盘上的问题：
- 每个节点存一个值 + 两个指针 → 可能只占几十字节
- 但一次 IO 要读一整页（16KB） → 浪费
- 树太高 → IO 次数多
```

```
例如 100 万条记录的 BST：
树高 ≈ log₂(1,000,000) ≈ 20 层
→ 查找一次最多 20 次磁盘 IO
→ 假设每次 10ms → 200ms！

能不能一次 IO 读更多数据，降低树高？
```

这就是 B+Tree。

## 核心特性

### "矮胖"的设计

```
BST（高瘦）：
        50
       /  \
     30    70
    / \   / \
   10 40 60 90
  /\
 5  15
...
树高 20 层，每层一个值

B+Tree（矮胖）：
        [30 | 50 | 70]           ← 每个节点存多个值
       /    |    |    \
    [5|10] [35|40] [55|60] [80|90]  ← 内部节点（非叶子）
     |  |    |  |    |  |    |  |
    [数据] [数据] [数据] [数据]     ← 叶子节点存数据

树高 2-3 层，每个节点一个磁盘页（16KB）
```

### B+Tree 的定义（m 阶）

```
1. 每个节点最多 m 个子节点
2. 每个内部节点最多 m-1 个 key
3. 每个内部节点最少 ⌈m/2⌉ 个子节点（根除外）
4. 所有叶子在同一层
5. 数据只存在叶子节点
6. 叶子节点之间有链表连接
```

### 为什么数据只存在叶子节点？

```
B-Tree（不是 B+Tree）：
数据可以存在内部节点
→ 查找可能在内层就结束

B+Tree：
数据只在叶子节点
→ 查找一定走到叶子
→ 但范围查询极快（叶子链表直接遍历）

InnoDB 选择 B+Tree 就是因为范围查询是主要场景
```

## 图解：页、节点、磁盘 IO

```
MySQL InnoDB 页大小：16KB（默认）

一个 B+Tree 内部节点（非叶子）：
┌──────────────────────────────────┐
│ Page Header                      │
├──────────────────────────────────┤
│ Key1 | PagePtr1 | Key2 | PagePtr2│  ← 索引项
├──────────────────────────────────┤
│ ...                              │
├──────────────────────────────────┤
│ KeyN | PagePtrN                  │
├──────────────────────────────────┤
│ Page Trailer (checksum)          │
└──────────────────────────────────┘

一个 B+Tree 叶子节点：
┌──────────────────────────────────┐
│ Pre | Next (链表指针)             │
├──────────────────────────────────┤
│ Key1 | RowData1                  │  ← 完整的行数据
├──────────────────────────────────┤
│ Key2 | RowData2                  │
├──────────────────────────────────┤
│ ...                              │
└──────────────────────────────────┘
```

```
查找过程：
SELECT * FROM users WHERE id = 888;

1. 读根节点（1次 IO）→ 确定子节点位置
2. 读子节点（1次 IO）→ 确定叶子位置
3. 读叶子节点（1次 IO）→ 找到数据

总共：3 次 IO（对于数百万行数据）

对比 BST：可能需要 20+ 次 IO
```

## 时间复杂度

| 操作 | B+Tree |
|------|--------|
| 等值查找 | O(log n) — 但常数很小（树极矮） |
| 范围查找 | O(log n + k) — k 是结果集大小 |
| 插入 | O(log n) |
| 删除 | O(log n) |

## 为什么不是 BST / 红黑树 / 数组？

### 为什么不是 BST？

```
BST 适合内存 → 树高，磁盘 IO 多
B+Tree 适合磁盘 → 树矮，一次 IO 读更多数据
```

### 为什么不是红黑树？

```
同样的问题 → 红黑树每个节点一个值 → 磁盘 IO 多
红黑树没有叶子链表 → 范围查询慢
```

### 为什么不是数组（二分查找）？

```
数组二分查找 → O(log n) + 支持范围
但插入数据需要移动所有后续元素 → O(n)

数据库需要频繁插入 → 数组不可行
```

## MySQL InnoDB 的 B+Tree

### 聚簇索引 (Clustered Index)

```
PRIMARY KEY 的 B+Tree：

叶子节点存的是「完整行数据」

         [10 | 20 | 30]        ← 内部节点：主键值 + 子页指针
        /    |    |    \
    [1..9] [11..19] [21..29] [31..]
      ↑
    叶子节点：完整行数据

查找 id=25：
1. 根节点：20 ≤ 25 < 30 → 去第 3 个子节点
2. 叶子：找到 id=25 的完整数据行
```

### 二级索引 (Secondary Index)

```
非主键索引的 B+Tree：

叶子节点存的是「主键值」

CREATE INDEX idx_name ON users(name);

B+Tree 按 name 排序：
叶子节点：name → primary_key

查找流程：
1. 在 idx_name 的 B+Tree 中找 name='Alice' → 得到 pk=25
2. 用 pk=25 去聚簇索引 → 得到完整行数据

这叫「回表」
```

### 覆盖索引 (Covering Index)

```sql
-- 需要回表（两次 B+Tree 查找）
SELECT * FROM users WHERE name = 'Alice';

-- 覆盖索引（不需要回表）
SELECT name, id FROM users WHERE name = 'Alice';
-- name 和 id 都在 idx_name 的叶子节点中
```

## 页的分裂与合并

```
插入导致页满 → 页分裂

分裂前：
[1,2,3,4,5] 满页，插入 6

分裂后：
[1,2,3] 和 [4,5,6] 两个半满页

极端情况：
[1,2,3] ← 插入 2.5 → 频繁分裂？

InnoDB 优化：
- 页填充因子：不填满，预留空间
- 顺序插入 → 不分裂（追加到最右）
- 随机插入 → 可能频繁分裂
```

```
为什么建议用自增主键？

自增主键 → 数据追加到最右 → 几乎不分裂 → 碎片少
UUID 主键 → 随机插入 → 频繁分裂 → 碎片多、性能差
```

---

## 小结

```
B+Tree = 为磁盘 IO 优化的平衡多路搜索树

设计哲学：
"不要做很多次小的 IO，做少量大的 IO"

BST → 每个节点一个值 → 高瘦 → 多次 IO
B+Tree → 每个节点多个值 → 矮胖 → 少量 IO

工程价值：
- MySQL InnoDB → 聚簇索引 + 二级索引
- 叶子链表 → 高效范围查询
- 自增主键 → 减少页分裂

核心洞察：
同一个数据结构，在不同存储介质上的表现完全不同。
B+Tree 不是比 BST 更好，而是更适合磁盘。
```
